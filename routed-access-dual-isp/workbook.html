Routed Access Campus with Dual ISPs

Hands-on lab for Cisco IOL-XE 17.18.2 and Containerlab.

Method: Build from outside inward. Configure one device section, run its checkpoint, then continue. Commands were selected for IOL-XE images used by this topology. Hardware forwarding, StackWise, EtherChannel hardware behavior, PoE, phone operation, and real ISP transit sit outside lab scope.

Learning objectives

By end of lab, you will be able to:

  • Read Containerlab-to-IOL interface mappings.
  • Build routed campus access with no Layer 2 uplinks.
  • Configure VLANs, SVIs, PortFast, and BPDU Guard at network edge.
  • Form OSPF point-to-point adjacencies with passive-by-default operation.
  • Divide OSPF into backbone Area 0 and totally stub Area 1.
  • Summarize access routes at distribution ABRs.
  • Use equal-cost OSPF paths for active/active uplinks.
  • Build eBGP to two simulated ISPs and loopback-sourced iBGP between borders.
  • Filter BGP routes with prefix lists and route maps.
  • Enforce maximum-prefix limits and customer source validation.
  • Translate private client addresses with border PAT.
  • Prefer ISP1 outbound with LOCAL_PREF and de-prefer ISP2 inbound with AS-path prepending.
  • Verify control plane, forwarding table, host reachability, and throughput.
  • Break links and diagnose expected convergence.

Lab rules and limits

  1. Do not configure Ethernet0/0. Containerlab reserves it for management.
  2. IOL groups ports by four. eth4 maps to Ethernet1/0, not Ethernet0/4.
  3. Save each IOS configuration after its checkpoint with write memory.
  4. Generic Ubuntu VMs see Containerlab eth1 as ens2 on this image. Verify with ip -br link.
  5. Host data interfaces use temporary Linux configuration. Redeploy removes it.
  6. internet-r1 is isolated simulated upstream. It has no host or real-Internet data-plane link.
  7. All public-looking addresses use RFC 5737 documentation blocks: 192.0.2.0/24, 198.51.100.0/24, and 203.0.113.0/24.
  8. Border PAT translates client 10.1.0.0/23 to active WAN interface address. ISP source ACLs reject untranslated RFC1918 and spoofed customer traffic.
  9. Shared upstream proves AS-path prepending: normal inbound path uses ISP1, then fails over to ISP2.
  10. Voice VLAN SVIs remain down until at least one active Layer 2 port belongs to each voice VLAN. This is expected IOS SVI autostate behavior.

Step 0 - Deploy and baseline

Isolation safety check

Topology data plane contains only explicit node-to-node links. internet-r1 is simulated upstream, not host uplink. Its default route terminates at Null0. Containerlab management bridge exists only for administration and is never used as routed transit.

Before configuration, verify IOL has no management-derived default:

show ip interface brief | include Ethernet0/0
show ip route 0.0.0.0

Expected state

Ethernet0/0 has Containerlab management address; default is not in routing table. Do not add NAT, OSPF, BGP, policy, or data-plane descriptions to Ethernet0/0.

From lab directory:

cd /opt/containerlab/labs/router-access-dual-isp
sudo clab deploy -t routed-access-dual-isp.clab.yml
sudo clab inspect -t routed-access-dual-isp.clab.yml

Connect to IOL nodes:

sudo clab connect -t routed-access-dual-isp.clab.yml --node border-r1

Connect directly by management name when SSH is ready:

ssh admin@clab-routed-access-dual-isp-border-r1

Baseline every IOS node:

show version
show interfaces brief
show cdp neighbors
show running-config | section router

Expected state

  • Image reports IOS XE 17.18.2.
  • Ethernet0/0 has management address.
  • Data interfaces have no lab IP configuration.
  • No OSPF or BGP process runs on clean deployment.

Destructive reset: sudo clab destroy -t routed-access-dual-isp.clab.yml -c removes containers and generated lab directory, including saved node state. Use only when starting lab again.

Device configuration

Step 1 - internet-r1

Open internet-r1 configuration and checkpoint

Goal

Create isolated upstream AS 64500. Advertise only default route to both ISPs. Accept only campus documentation prefix. Shared upstream makes inbound path selection observable.

Configure

enable
configure terminal
hostname internet-r1
ip routing

interface Loopback0
 description SIMULATED-PUBLIC-SERVICE
 ip address 192.0.2.200 255.255.255.255
 exit

interface Ethernet0/1
 description TO-ISP1
 ip address 192.0.2.6 255.255.255.252
 no shutdown
 exit

interface Ethernet0/2
 description TO-ISP2
 ip address 198.51.100.6 255.255.255.252
 no shutdown
 exit

ip route 0.0.0.0 0.0.0.0 Null0
ip route 192.0.2.0 255.255.255.252 192.0.2.5
ip route 198.51.100.0 255.255.255.252 198.51.100.5

ip prefix-list CAMPUS-IN seq 5 permit 203.0.113.0/24
ip prefix-list TRANSIT-OUT seq 5 permit 0.0.0.0/0

route-map CAMPUS-IN permit 10
 match ip address prefix-list CAMPUS-IN
 exit

route-map TRANSIT-OUT permit 10
 match ip address prefix-list TRANSIT-OUT
 exit

router bgp 64500
 bgp router-id 192.0.2.200
 bgp log-neighbor-changes
 neighbor 192.0.2.5 remote-as 65001
 neighbor 192.0.2.5 description ISP1
 neighbor 192.0.2.5 route-map CAMPUS-IN in
 neighbor 192.0.2.5 route-map TRANSIT-OUT out
 neighbor 198.51.100.5 remote-as 65002
 neighbor 198.51.100.5 description ISP2
 neighbor 198.51.100.5 route-map CAMPUS-IN in
 neighbor 198.51.100.5 route-map TRANSIT-OUT out
 address-family ipv4
  network 0.0.0.0 mask 0.0.0.0
  neighbor 192.0.2.5 activate
  neighbor 192.0.2.5 maximum-prefix 1
  neighbor 198.51.100.5 activate
  neighbor 198.51.100.5 maximum-prefix 1
  exit-address-family
 exit
end
write memory

Checkpoint

show ip interface brief
show ip route static
show ip route 192.0.2.2
show ip route 198.51.100.2
show ip bgp summary
show ip bgp 203.0.113.0
show ip bgp neighbors 192.0.2.5 advertised-routes

Expected state

upstream interfaces and local default work. BGP peers wait for ISP configuration.

Open isp1 configuration and checkpoint

Goal

Create ISP AS 65001 with customer and upstream eBGP. Accept campus prefix only from customer, accept default and public test route only from upstream, then export each in correct direction.

Configure

interface Loopback0
 ip address 192.0.2.101 255.255.255.255

 description TO-BORDER-R1

 no shutdown
 exit

interface Ethernet0/2
 description TO-INTERNET-R1
 ip address 192.0.2.5 255.255.255.252
 no shutdown
 exit

ip prefix-list CUSTOMER-IN seq 5 permit 203.0.113.0/24
ip prefix-list CUSTOMER-OUT seq 5 permit 0.0.0.0/0
ip prefix-list UPSTREAM-IN seq 5 permit 0.0.0.0/0
ip prefix-list UPSTREAM-OUT seq 5 permit 203.0.113.0/24

route-map CUSTOMER-OUT permit 10
 match ip address prefix-list CUSTOMER-OUT
 exit
route-map UPSTREAM-IN permit 10
 match ip address prefix-list UPSTREAM-IN
 exit
route-map UPSTREAM-OUT permit 10
 match ip address prefix-list UPSTREAM-OUT
 exit

router bgp 65001
 bgp router-id 192.0.2.101
 bgp log-neighbor-changes
 neighbor 192.0.2.2 remote-as 65000
 neighbor 192.0.2.2 description CAMPUS-BORDER-R1
 neighbor 192.0.2.2 route-map CUSTOMER-IN in
 neighbor 192.0.2.2 route-map CUSTOMER-OUT out
 neighbor 192.0.2.6 remote-as 64500

 neighbor 192.0.2.6 route-map UPSTREAM-OUT out

  neighbor 192.0.2.6 activate
  neighbor 192.0.2.6 maximum-prefix 1

 exit

Step 4 - border-r1

Open border-r1 configuration and checkpoint

Goal

Join OSPF Area 0, peer with ISP1 using eBGP, peer with border-r2 using loopback-sourced iBGP, prefer ISP1 default with LOCAL_PREF 200, advertise only campus public test prefix, and PAT clients.

Build interfaces and OSPF

enable
configure terminal
hostname border-r1
ip routing

interface Loopback0
 ip address 10.0.255.1 255.255.255.255
 ip ospf 1 area 0
 exit

interface Ethernet0/1
 description EBGP-TO-ISP1
 ip address 192.0.2.2 255.255.255.252
 ip nat outside
 no shutdown
 exit

interface Ethernet0/2
 description IBGP-TRANSIT-TO-BORDER-R2
 ip address 10.255.0.1 255.255.255.252
 ip nat inside
 ip ospf 1 area 0
 ip ospf network point-to-point
 no shutdown
 exit

interface Ethernet0/3
 description TO-CORE-R1
 ip address 10.0.0.1 255.255.255.252
 ip nat inside
 ip ospf 1 area 0
 ip ospf network point-to-point
 no shutdown
 exit

interface Ethernet1/0
 description TO-CORE-R2
 ip address 10.0.0.5 255.255.255.252
 ip nat inside
 ip ospf 1 area 0
 ip ospf network point-to-point
 no shutdown
 exit

router ospf 1
 router-id 10.0.255.1
 passive-interface default
 no passive-interface Ethernet0/2
 no passive-interface Ethernet0/3
 no passive-interface Ethernet1/0
 default-information originate
 exit
end

Design reason

  • Point-to-point removes DR/BDR election from two-node routed links.
  • Passive default advertises loopback but sends hellos only toward known routers.
  • Conditional origination injects OSPF default only while routing table contains a default.

Build BGP policy and peers

configure terminal
ip access-list standard NAT-SOURCES
 permit 10.1.0.0 0.0.1.255
 exit
ip nat inside source list NAT-SOURCES interface Ethernet0/1 overload

ip route 203.0.113.0 255.255.255.0 Null0

ip prefix-list ISP1-ACCEPT seq 5 permit 0.0.0.0/0
ip prefix-list CAMPUS-PREFIX seq 5 permit 203.0.113.0/24

route-map ISP1-IN permit 10
 match ip address prefix-list ISP1-ACCEPT
 set local-preference 200
 exit

route-map CAMPUS-OUT permit 10
 match ip address prefix-list CAMPUS-PREFIX
 exit

router bgp 65000
 bgp router-id 10.0.255.1
 bgp log-neighbor-changes
 neighbor 192.0.2.1 remote-as 65001
 neighbor 192.0.2.1 description ISP1
 neighbor 192.0.2.1 route-map ISP1-IN in
 neighbor 192.0.2.1 route-map CAMPUS-OUT out
 neighbor 10.0.255.2 remote-as 65000
 neighbor 10.0.255.2 description IBGP-BORDER-R2
 neighbor 10.0.255.2 update-source Loopback0
 neighbor 10.0.255.2 next-hop-self
 address-family ipv4
  network 203.0.113.0 mask 255.255.255.0
  neighbor 192.0.2.1 activate
  neighbor 192.0.2.1 maximum-prefix 1
  neighbor 10.0.255.2 activate
  exit-address-family
 exit
end
write memory

Checkpoint

show ip ospf interface brief
show ip ospf neighbor
show ip bgp summary
show ip bgp neighbors 192.0.2.1 routes
show ip bgp neighbors 192.0.2.1 advertised-routes
show ip bgp 0.0.0.0
show ip prefix-list
show route-map
show ip nat statistics
show ip nat translations

Expected state

  • ISP1 eBGP reaches Established and default has local preference 200.
  • iBGP waits until border-r2 loopback becomes reachable through OSPF.
  • Outbound advertisement contains 203.0.113.0/24 only.
  • No OSPF neighbor forms on ISP-facing Ethernet0/1.

Check your understanding: Why is conditional default origination safer than always?

Research notes

Design choices align with official documentation:

  • RFC 5737 reserves 192.0.2.0/24, 198.51.100.0/24, and 203.0.113.0/24 for documentation. These blocks should not appear on public Internet. Source: https://www.rfc-editor.org/info/rfc5737/
  • Cisco IOS XE OSPF guide: stub areas suppress external LSAs; no-summary suppresses inter-area summaries; area range summarizes contiguous prefixes at ABR. Source: https://www.cisco.com/c/en/us/td/docs/routers/ios/config/17-x/ip-routing/b-ip-routing/m_iro-cfg-0.html
  • Cisco IOS XE passive-interface guide: passive interfaces advertise connected networks without sending OSPF hellos or forming neighbors. Source: https://www.cisco.com/c/en/us/td/docs/routers/ios/config/17-x/ip-routing/b-ip-routing/m_iri-default-passive-interface.html
  • Cisco IOS XE external BGP guide: prefix filters and route maps constrain advertisements; LOCAL_PREF controls outbound preference inside AS; AS-path prepend influences remote selection when upstream topology provides alternate paths. Source: https://www.cisco.com/c/en/us/td/docs/routers/ios/config/17-x/ip-routing/b-ip-routing/m_irg-external-sp-0.html
  • Cisco IOS XE maximum-prefix guide: per-neighbor limits protect BGP sessions from excessive routes. Source: https://www.cisco.com/c/en/us/td/docs/routers/ios/config/17-x/ip-routing/b-ip-routing/m_irg-max-prefix.html
  • Cisco PortFast and BPDU Guard guidance: use both only on end-station-facing ports. Source: https://www.cisco.com/c/en/us/support/docs/lan-switching/spanning-tree-protocol/10586-65.html
  • Containerlab deploy reference: -t selects topology; reconfigure performs destructive clean recreation. Source: https://containerlab.dev/cmd/deploy/
